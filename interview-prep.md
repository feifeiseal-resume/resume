## 1. 核心定位

這次面試的主軸要很清楚：我是前端工程師。需求理解能力要從過去參與前端專案規劃、需求釐清與技術討論的經驗自然帶出，不需要特別強調以前的 PM 經歷。

> 我是一位有近四年前端實務經驗的工程師，主力是 Vue 3、Nuxt 3、TypeScript、RWD、API 串接與前端效能 / SEO 優化。過去在企業系統、品牌官網和 SaaS 產品開發中，我經常參與需求釐清和技術討論，因此習慣從前端角度理解使用流程、資料狀態和實作風險，再把需求落成可維護、可擴充、使用體驗好的產品。

面試時要讓對方記住三件事：

- 我有實際前端開發經驗：Vue / Nuxt / TypeScript、企業系統、官網、SaaS、RWD、API 串接。
- 我有產品品質意識：Component 化、資料狀態、錯誤處理、Code Review、技術債和維護性。
- 我會使用 AI，但不依賴 AI：Cursor 可以加速理解、實作、Debug 和文件整理，最後仍由我檢查架構、型別、邏輯和邊界情境。

一句話定位：

> 我不是只會切畫面的前端，我會從需求、使用流程、元件結構、資料狀態和後續維護一起思考，然後把它落成穩定、好維護的產品。

## 3. 必問題準備

### Q1. 為什麼想來我們公司？

我想找一個重視產品品質、前端細節和團隊協作的環境。從職缺描述看起來，貴公司很在意網站效能、SEO、UX、Component 化、Code Review、技術債改善，以及 AI 輔助開發，這些都和我過去做企業系統、Nuxt 官網和 SaaS 產品時累積的經驗很接近。

我也很認同 JD 裡提到的「看到設計稿會想讓它更順、更快、更漂亮」這種前端心態。因為我自己在做官網和系統介面時，也會在意畫面是否符合設計、互動是否順、資料狀態是否清楚，以及程式後續是否容易維護。

如果從官網和職缺描述來看，這個職位不像只是做一次性的頁面，而是需要長期維護、優化體驗和持續迭代產品，這點也是我想繼續累積的方向。

### Q2. 你怎麼使用 AI 工具？

我目前主要使用 Cursor 協助日常開發，常見情境包含理解既有程式碼、拆解需求、整理實作方向、產生初版程式碼、協助 Debug，以及檢查有沒有遺漏的邊界情境。

我也會用 AI 協助產生技術文件或開發紀錄，例如整理元件使用方式、實作步驟和問題排查過程。對我來說，AI 可以提升探索和實作速度，但它不應該取代工程師的判斷。

我通常會把 AI 當成加速理解和產出初稿的工具，但架構選擇、資料流、命名、型別設計和是否符合專案風格，還是會由我自己判斷。

### Q3. AI 產出的程式你怎麼檢查？

我會先回到需求本身，確認功能是否真的符合使用情境，再檢查程式邏輯、資料流、狀態管理和錯誤處理是否合理。

接著我會看程式是否符合專案既有風格，例如命名、元件拆分方式、API 串接方式和型別定義。如果是比較重要的功能，我會特別檢查邊界情境，例如空資料、載入中、錯誤狀態、權限限制，以及不同裝置上的呈現。

如果 AI 產出測試，我也會確認測試是否真的對應需求，而不是只測到表面。例如表單驗證要看必填、格式錯誤、API 失敗、送出中狀態；元件測試要看 props、事件、條件渲染和空資料狀態。

### Q4. 你有沒有 Code Review 經驗？

有。過去在專案中，我曾參與既有 codebase 維護、模組化改寫和共用元件庫建置，也有和技術長、前端同事、後端工程師一起討論實作方式。

我在 Code Review 時會看幾件事：需求有沒有被正確完成、程式是否容易理解和維護、元件和狀態管理是否拆得合理、有沒有潛在 bug 或邊界情境沒有處理，以及有沒有不必要的重構或過度設計。

對我來說，Code Review 不只是挑錯，而是讓團隊對程式品質、實作方向和維護方式有共同理解。

### Q5. 你對測試、Docker、CI 熟不熟？

我過去比較多是參與功能測試、UAT 驗收和手動驗證流程，會根據需求確認功能、資料狀態、權限和畫面互動是否符合預期。

單元測試的部分，我目前正在補強 Vitest、Vue Test Utils 這類前端測試工具。因為這份職缺也有提到 AI 撰寫測試，我會希望把 AI 當成輔助工具，先產生測試案例，再由我確認測試是否真的覆蓋需求和邊界情境。

Docker 和 CI 的部分，我具備基本概念，理解它們在環境一致性、自動化檢查、建置和部署流程中的用途。過去雖然不是主要維護 CI/CD 的角色，但如果進到團隊，我會先從前端最直接相關的 lint、type check、build、test 問題開始熟悉，確保自己能先把日常 PR 品質顧好。

### Q6. 你看到設計稿時，會怎麼開始前端開發？

我會先把設計稿當成「產品介面規格」來看，而不是只把它當成切版圖。第一步會先看整體版面結構和共用規則，例如 spacing、字級、顏色、按鈕、表單、卡片、列表、彈窗和互動區塊是否能抽成共用元件。

第二步會確認設計稿裡沒有明講、但前端一定會遇到的狀態，例如 RWD 斷點、hover / active / disabled、loading、空資料、錯誤狀態、文字過長、圖片比例、表單驗證和 API 失敗。如果設計稿沒有定義清楚，我會主動整理問題和建議做法，再和設計或 PM 確認。

實作時我會先拆 component 和 layout，再處理資料流、API 串接與狀態管理。完成後除了確認視覺還原，也會檢查不同瀏覽器和裝置上的呈現、互動是否順、程式是否有重複，以及未來新增相似頁面時是否容易擴充。

### Q7. 你如何處理設計稿和實際前端開發之間的落差？

我覺得設計稿轉成前端時，最重要的是不要只追求靜態畫面一樣，而是要把它轉成真實可使用、可維護的介面。實務上常遇到的落差包含：資料長度不固定、API 載入時間、錯誤狀態、不同螢幕尺寸、瀏覽器差異，以及設計稿沒有畫到的互動狀態。

遇到這些情況，我會先判斷哪些可以由前端直接做合理處理，例如文字截斷、骨架 loading、空狀態提示、圖片比例限制；哪些需要和設計或 PM 確認，例如行動版排版大幅改變、資訊優先順序、操作流程調整。我的目標是讓畫面接近設計，也讓實際使用情境是順的。

### Q8. 你怎麼看待重構？

我不害怕重構，但我會先判斷重構的目的和成本。如果是會影響開發效率、容易產生 bug，或已經阻礙新功能擴充的技術債，我會支持重構。

但如果只是個人偏好，或短期內風險大於收益，我會傾向先局部改善。我比較習慣把重構控制在可驗證的範圍內，例如先整理元件責任、改善命名、移除重複邏輯、補上型別或測試，再逐步處理較大的架構問題。
