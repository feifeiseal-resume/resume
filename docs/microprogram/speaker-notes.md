## 5. JOSUI：問題與產品定位｜1:40

第一個想分享的是 JOSUI。它是一個為 shadcn/ui 和 Tailwind CSS 做的主題色彩工具。

這個專案的起點，其實是我觀察到設計和開發在對齊色彩 token 的時候，常常需要來回確認。即使只是調整一個主題色，也不一定能很快看出整體元件會怎麼變化。

所以我就把這個問題做成一個工具。使用者可以直接在瀏覽器裡調整 token、即時看元件效果，最後再把整理好的色彩設定帶回自己的專案。

對我來說，這個專案不只是練習 React，而是一次完整的小型產品實驗：從問題定義、互動設計、元件實作，到最後部署到 Cloudflare，都是我自己完成的。

## 6. JOSUI：技術實作｜1:20

技術上，我希望這個預覽是真的會跟著 token 變動，而不是只做一個看起來像預覽的靜態畫面。

所以我把主題色彩的真實來源放在 CSS custom properties，透過 CSSOM 去讀寫 `:root` 和 `.dark` 裡面的 variables。React 主要負責面板開關、目前選到的 preset 這些 UI 狀態。

這樣一來，使用者調整色彩時，shadcn/ui 元件會馬上更新，也可以匯出 Tailwind v3 或 v4 的 token。

這段實作讓我比較有感的是，工具型產品其實很需要先想清楚資料來源和責任邊界。這些先整理好，後面的功能才比較容易維護和延伸。

## 7. JOSUI：AI 協作｜0:55

JOSUI 也是我實際把 AI 放進開發流程的專案。AI 有協助過需求和元件設計的發想、React 和 TypeScript 的初稿、樣式調整，也包含部署和文件整理。不過我最常使用它的地方，其實是除錯和問題分析。

例如問題牽涉到元件狀態、樣式呈現，或主題 token 行為時，我會先整理一個比較小的重現情境，再請 AI 幫我一起看可能原因和排查方向。

但最後我還是會回到程式碼和畫面本身，確認型別、資料流、元件邊界，以及實際結果是不是符合預期。

所以對我來說，AI 比較像是加速分析和驗證的協作工具。它可以幫我更快縮小問題範圍，但不會取代工程師自己的判斷。

## 8. 都市探針 City Probe：專案概覽｜1:35

接下來是我在勝崨參與的都市探針 City Probe。這是一個配合大學學術研究開發的智慧城市市民參與平台。

前端的範圍滿廣的，包含登入認證、權限控制、資料表格、儀表板、地圖和多語系等功能。

我主要負責 Nuxt 3 的頁面和元件開發、API 串接與狀態管理，也有處理使用者資料、權限，以及表單、資料查詢和地圖相關功能。

都市探針讓我第一次比較具體地接觸到智慧城市的資料應用。也讓我發現，前端在真實場域裡不只是做介面而已，它其實也是協助使用者理解資訊、完成操作，甚至做出決策的一環。

## 9. 都市探針 City Probe：技術挑戰｜1:35

這個專案裡，我印象比較深的是 Nuxt SSR 和地圖套件之間的邊界處理。

因為地圖相關套件通常需要瀏覽器環境才能執行，如果直接在伺服器端載入，就可能會出錯。所以我會把地圖和瀏覽器限定的套件放在 client-only 的環境裡處理，也透過 Nuxt plugins 去管理表格、tooltip、日期選擇這類瀏覽器端套件。

狀態管理方面，我使用 Pinia 管理使用者、語系和主題等跨頁共用資訊，也用 TypeScript 約束 API 資料和元件 props。

這個專案讓我更理解，功能變複雜時，架構不是為了把檔案拆開而已，而是要讓後續維護和擴充都有清楚的邊界。

## 10. FUKUKU 官網｜1:00

FUKUKU 是一個 Nuxt 3 官網專案。這個專案是由設計師先和業主確認設計稿，我這邊主要負責把確認後的設計，轉成可維護的前端架構、頁面元件和視覺實作。

這個專案的重點不只是把畫面還原。官網還需要考慮搜尋曝光、跨裝置閱讀和載入效率，所以我使用 Nuxt 3 SSR 處理 SEO，搭配 RWD 讓不同裝置上都能維持清楚的閱讀體驗，也透過圖片 Lazy Loading 和 Code Splitting 降低初次載入負擔。

這個案例讓我更熟悉，怎麼在設計呈現、品牌需求和工程實作之間，做出比較合適的取捨。
